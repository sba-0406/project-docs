Processes & Threads

One of the most asked OS topics in backend interviews. Understand the difference cold — interviewers ask this even at mid-tier companies.

01 What is a Process?
Must Know
✓
▼
Definition

A process is a program in execution. When you run a program, the OS creates a process for it — it gets its own memory space (code, data, heap, stack), a process ID, and system resources.

Key Properties
→Each process has its own separate memory space — one process cannot directly access another's memory
→Process creation is heavy — the OS must allocate new memory and resources (fork() is expensive)
→Processes communicate via IPC: pipes, sockets, shared memory, message queues
→A process can have multiple threads inside it
Process States
→New — process is being created
→Ready — waiting to be assigned to CPU
→Running — currently being executed by CPU
→Waiting/Blocked — waiting for I/O or event
→Terminated — finished execution
How to say it in an interview

"A process is an instance of a running program. The OS isolates each process with its own memory space — code segment, data segment, heap, and stack. This isolation is why if one process crashes, others aren't affected."

Likely Interview Questions
What is the difference between a program and a process?
What happens when you fork a process?
How do processes communicate with each other?
02 What is a Thread?
Must Know
✓
▼
Definition

A thread is the smallest unit of execution within a process. Multiple threads within the same process share code, data, and heap — but each thread has its own stack and registers.

Key Properties
→Threads are lightweight — creating a thread is faster than creating a process
→Threads within the same process share memory — so they can communicate directly (but need synchronisation)
→If one thread crashes badly (segfault), it can bring down the entire process
→Java uses threads heavily — every Java program has at least the main thread
Process vs Thread — The Table You Must Know
AspectProcessThread
MemorySeparate memory spaceShared memory (within process)
Creation costHeavy (slow)Lightweight (fast)
CommunicationIPC (pipes, sockets)Shared variables (need locks)
IsolationCrash doesn't affect othersCrash can kill entire process

ExampleChrome browser itselfEach Chrome tab's JS engine
How to say it in an interview

"A thread is a lightweight unit of execution inside a process. Threads share the same memory space, which makes communication easy but requires careful synchronisation to avoid race conditions. Node.js is single-threaded for JS execution but uses worker threads via libuv for I/O."

03 Context Switching
Important
✓
▼
Definition

Context switching is when the CPU stops executing one process/thread and starts executing another. The OS saves the current state (registers, program counter, stack pointer) of the running process into its PCB, then loads the saved state of the next process.

Why it matters
→Context switching has overhead — saving and restoring state takes time
→Too many context switches = "thrashing" — CPU spends more time switching than doing work
→This is why Node.js's event loop is efficient — avoids context switching by using async I/O on a single thread
How to say it in an interview

"Context switching is the OS's way of giving each process a turn on the CPU. The current state is saved in the PCB (Process Control Block) and the next process's state is restored. It has overhead, which is one reason event-driven architectures like Node.js are efficient — they avoid the cost of switching between threads."

Likely Interview Questions

CPU Scheduling

Know the algorithms and their tradeoffs. Common in OS theory rounds at mid-tier companies.

01 Scheduling Algorithms
Must Know
✓
▼
What is CPU Scheduling?

The OS decides which process/thread gets CPU time and for how long. The goal is to maximise CPU utilisation, throughput, and minimise waiting time and response time.

Key Algorithms
AlgorithmHow it worksProblem
FCFSFirst come, first served — simple queueConvoy effect — short jobs wait behind long ones
SJFShortest job runs firstStarvation — long jobs may never run
Round RobinEach process gets a fixed time slice (quantum), then goes back to queueIf quantum too small = too many context switches
PriorityHigher priority process runs firstStarvation of low-priority processes
MLFQMultiple queues with different priorities. Process demoted if it uses too much CPUComplex to implement
Key Terms
→Burst time — time needed by process on CPU
→Waiting time — time spent in ready queue
→Turnaround time — total time from submission to completion
→Preemptive — OS can yank CPU from a running process (Round Robin)
→Non-preemptive — process runs until it finishes or blocks (FCFS)
How to say it in an interview

"Round Robin is the most practical scheduling algorithm — each process gets equal CPU time in turns. It's preemptive and fair, but if the time quantum is too small, context switching overhead dominates. Modern OSes use MLFQ — processes start with high priority but get demoted if they're CPU-intensive."

Memory Management

Paging, virtual memory, and memory allocation. Backend engineers get asked this to test OS fundamentals.

01 Paging & Virtual Memory
Must Know
✓
▼
Virtual Memory

Virtual memory gives each process the illusion of having its own large, contiguous memory space — even if physical RAM is limited. The OS uses disk as an extension of RAM (swap space).

Paging
→Physical memory is divided into fixed-size blocks called frames
→Process memory is divided into same-size blocks called pages
→Page Table maps virtual page numbers to physical frame numbers
→No external fragmentation (pages fit perfectly into frames)
→Page fault — when a page isn't in RAM, OS fetches it from disk (slow!)
Stack vs Heap (Memory Layout)
StackHeap
Fixed size, managed automaticallyDynamic size, manually managed (or GC)
Stores local variables, function callsStores objects, dynamically allocated memory
LIFO orderNo specific order
Fast accessSlower (allocation overhead)
Stack overflow if too deep recursionMemory leak if not freed
How to say it in an interview

"Virtual memory abstracts physical RAM — each process sees a large virtual address space. The OS uses paging to map virtual pages to physical frames. When a page isn't in RAM, a page fault occurs and the OS fetches it from swap — which is why excessive swapping (thrashing) kills performance."

Deadlocks

Very commonly asked. Know the 4 conditions and how to prevent/detect deadlocks.

01 What is a Deadlock?
Frequently Asked
✓
▼
Definition

A deadlock is a situation where two or more processes are stuck waiting for each other to release resources — and none of them ever will. Everyone is waiting, nothing moves.

The 4 Coffman Conditions (ALL must be true for deadlock)
1Mutual Exclusion — only one process can use a resource at a time
2Hold and Wait — a process holds one resource while waiting for another
3No Preemption — resources cannot be forcibly taken from a process
4Circular Wait — P1 waits for P2, P2 waits for P3, P3 waits for P1
Prevention vs Avoidance vs Detection
ApproachHow
PreventionEliminate one of the 4 conditions (e.g., force processes to request all resources at once → breaks Hold and Wait)
AvoidanceBanker's Algorithm — only grant resource if system remains in "safe state"
DetectionAllow deadlock, detect using Resource Allocation Graph, then recover (kill a process)
How to say it in an interview

"A deadlock requires all 4 Coffman conditions simultaneously. In practice, you prevent deadlocks by imposing a global ordering on resource acquisition — for example, always lock mutex A before mutex B. This breaks circular wait. In databases, deadlock detection + transaction rollback is the standard approach."

Likely Interview Questions
What are the 4 necessary conditions for a deadlock?
How does the Banker's Algorithm work?
How do databases handle deadlocks?
What is livelock? How is it different from deadlock?
Quick Memory Aid
Livelock = processes keep responding to each other but make no progress (like two people in a corridor both stepping aside in the same direction)
Starvation = a process waits indefinitely because others keep getting priority
Deadlock = everyone is stuck waiting, nothing happens

Synchronisation

Mutex, semaphore, monitors — these come up when discussing concurrent programming in Node.js or Java interviews.

01 Mutex vs Semaphore vs Monitor
Frequently Asked
✓
▼
The Core Problem

When multiple threads access shared resources simultaneously, you get race conditions — unpredictable results because the outcome depends on the order of execution. Synchronisation tools prevent this.

ToolWhat it doesKey difference
MutexBinary lock — only one thread can hold it at a time. The same thread that locks it must unlock it.Ownership — only the locking thread can unlock
SemaphoreCounter-based — allows N threads to access a resource simultaneously. Can be signalled by a different thread.No ownership concept
MonitorHigh-level construct combining a mutex + condition variables. Java's synchronized keyword creates a monitor.Built-in in Java
SpinlockThread actively polls ("spins") waiting for lock. Fast for very short waits, wastes CPU otherwise.Busy-waiting
Critical Section Problem
→Critical Section — code that accesses shared resources, must be executed by only one thread at a time
→Mutual Exclusion — only one thread in critical section at a time
→Progress — if no thread is in critical section, one waiting thread should get in
→Bounded Waiting — a thread must eventually get to enter (no starvation)
How to say it in an interview

"A mutex is for mutual exclusion — only the thread that acquired it can release it. A semaphore is more flexible — it's a counter, and any thread can signal it. In Java, the synchronized keyword implicitly uses a monitor, which combines a mutex with wait/notify for condition-based coordination."

ACID & Transactions

Every backend interview will touch this. You used PostgreSQL transactions — be ready to explain exactly what ACID means.

01 ACID Properties
Must Know
✓
▼
What is a Transaction?

A transaction is a sequence of database operations that must be executed as a single logical unit. Either ALL operations succeed, or NONE of them do. Example: transferring money — deduct from A AND credit to B must both happen or neither should.

ACID — Each Property Explained
AAtomicity — "All or nothing." If any part of the transaction fails, the entire transaction is rolled back. No partial updates. Uses ROLLBACK to undo changes.
CConsistency — The database moves from one valid state to another. All integrity constraints (foreign keys, unique constraints) must hold before and after the transaction.
IIsolation — Concurrent transactions should not interfere with each other. Each transaction should execute as if it's the only one running. Implemented via locks or MVCC.
DDurability — Once a transaction is committed, it's permanent — even if the system crashes. Achieved via write-ahead logging (WAL) — changes are written to log before applying to database.
Isolation Levels (PostgreSQL)
LevelDirty ReadNon-Repeatable ReadPhantom Read
Read Uncommitted✓ Possible✓ Possible✓ Possible
Read Committed (default PG)✗ Prevented✓ Possible✓ Possible
Repeatable Read✗✗ Prevented✓ Possible
Serializable✗✗✗ Prevented
How to say it in an interview

"ACID ensures reliable transaction processing. Atomicity handles failure — if one query in a transaction fails, everything rolls back. Isolation prevents dirty reads between concurrent transactions — PostgreSQL uses MVCC (Multi-Version Concurrency Control) so readers don't block writers. Durability is achieved via WAL — PostgreSQL writes changes to a log before applying them."

Normalisation

Know 1NF through 3NF and why we normalise. Denormalisation is also important for system design.

01 Normal Forms (1NF → 3NF)
Must Know
✓
▼
Why Normalise?

To eliminate data redundancy and update anomalies. If the same data is stored in multiple places, updating it in one place but not others causes inconsistency.

Normal Forms
1NFFirst Normal Form — Each column has atomic (indivisible) values. No repeating groups. Each row is unique (has a primary key). Bad: storing "Math, Science" in one cell. Good: separate rows.
2NFSecond Normal Form — Must be in 1NF + No partial dependency. Every non-key column must depend on the WHOLE primary key (not part of it). Only applies to composite keys.
3NFThird Normal Form — Must be in 2NF + No transitive dependency. Non-key columns must depend ONLY on the primary key — not on other non-key columns.
BCNFBoyce-Codd NF — Stricter version of 3NF. For every functional dependency X → Y, X must be a superkey.
Anomalies normalisation prevents
→Insertion anomaly — can't insert data without other data being present
→Update anomaly — changing data in one row creates inconsistency in others
→Deletion anomaly — deleting a row unintentionally destroys other data
How to say it in an interview

"Normalisation removes redundancy. 1NF means atomic values and unique rows. 2NF removes partial dependencies (each attribute depends on the full primary key). 3NF removes transitive dependencies (attributes depend only on the primary key). In practice, we often denormalise for read performance — adding redundant columns to avoid expensive joins."

Likely Interview Questions

Indexing

Critical for backend interviews. You used MongoDB and PostgreSQL — be ready to explain how indexes work in both.

01 How Database Indexes Work
Must KnowFrequently Asked
✓
▼
What is an Index?

An index is a separate data structure that stores a sorted subset of a table's data, along with pointers to the actual rows. It allows the database to find rows without scanning the entire table — like a book's index vs reading every page.

B-Tree Index (Default in PostgreSQL & MySQL)
→Data stored in a balanced tree — O(log n) search time
→Supports range queries: WHERE age BETWEEN 20 AND 30
→Supports ORDER BY efficiently
→Default index type in PostgreSQL
Types of Indexes
TypeWhen to use
Primary IndexAutomatically created on primary key. Table is physically sorted by this.
Secondary IndexOn non-primary columns. Email, username etc.
Composite IndexOn multiple columns. WHERE city = 'HYD' AND age = 22. Order matters!
Unique IndexEnforces uniqueness on a column
Full-text IndexFor text search queries
Hash IndexExact match only. Faster than B-tree for equality checks, no range support
When NOT to index
→Columns with low cardinality (e.g., boolean, gender) — index won't help much
→Tables with very few rows — full scan may be faster
→Columns that are frequently updated — index must be updated too (overhead)
How to say it in an interview

"An index is a B-tree data structure maintained alongside the table. When you query WHERE email = 'x@y.com', instead of a full table scan (O(n)), the database traverses the B-tree in O(log n). The tradeoff is write overhead — every INSERT/UPDATE/DELETE must also update the index. In MongoDB, indexes work the same way — I always index fields used in find() queries and the aggregation pipeline's $match stage."

CAP Theorem

System design staple. You'll be asked this when discussing distributed database choices.

01 CAP Theorem Explained
Must Know
✓
▼
CAP Theorem

In a distributed system, you can only guarantee 2 of these 3 properties at the same time: Consistency, Availability, and Partition Tolerance.

The 3 Properties
CConsistency — Every read receives the most recent write or an error. All nodes see the same data at the same time.
AAvailability — Every request receives a response (not necessarily the latest data). System always stays online.
PPartition Tolerance — System continues to operate even if network partitions occur (nodes can't talk to each other). In real distributed systems, P is always required.
Since P is required, you choose: CP or AP
ChoiceExampleTradeoff
CP (Consistent + Partition Tolerant)MongoDB (in default config), HBase, ZookeeperMay become unavailable during partition
AP (Available + Partition Tolerant)Cassandra, CouchDB, DynamoDBMay return stale data during partition
CA (Consistent + Available)Traditional RDBMS like PostgreSQL (single node)Can't handle network partitions — only for single-node
How to say it in an interview

"CAP theorem says in a distributed system, you can't have all three — C, A, and P — simultaneously. Since network partitions are unavoidable in distributed systems, P is always chosen. So the real choice is between CP and AP. For a banking system, I'd choose CP — I'd rather return an error than return wrong balance data. For a social media feed, AP is fine — showing a slightly stale post is acceptable."

4 Pillars of OOP

Asked in every single technical interview. Know each pillar with a real code example.

01 Encapsulation
Must Know
✓
▼
Definition

Bundling data (fields) and methods that operate on that data into a single unit (class), and restricting direct access to some components using access modifiers.

→Private fields can only be accessed via public getters/setters
→Prevents external code from putting the object in an invalid state
→Information hiding — expose only what's necessary
// Java example public class BankAccount { private double balance; // private — cannot be accessed directly public double getBalance() { return balance; } // controlled access public void deposit(double amount) { if (amount > 0) balance += amount; // validation inside } } // You can't do: account.balance = -1000; (compile error)
How to say it in an interview

"Encapsulation bundles data and behaviour and hides implementation details. The key benefit is data integrity — users of a class can't put it in an invalid state. In my Node.js code, I encapsulate business logic inside service classes — the controller doesn't know HOW a job is scored, just that it calls assessmentService.scoreCandidate()."

02 Inheritance
Must Know
✓
▼
Definition

A child class inherits properties and methods from a parent class, enabling code reuse. The child can override parent methods and add new ones.

class Animal { String name; void speak() { System.out.println("..."); } } class Dog extends Animal { @Override void speak() { System.out.println("Woof!"); } // override void fetch() { System.out.println("Fetching!"); } // new method }
Types of Inheritance in Java
→Single — A extends B
→Multilevel — A extends B extends C
→Hierarchical — B extends A, C extends A
→Multiple — NOT supported in Java for classes (but supported via interfaces)
How to say it in an interview

"Java doesn't support multiple class inheritance to avoid the diamond problem — where two parent classes have the same method and the child doesn't know which to use. Java solves this with interfaces, which support multiple inheritance."

SOLID Principles

Senior engineers love asking SOLID. Knowing these shows you write production-quality code.

01 SOLID — All 5 Principles
Important
✓
▼
SSingle Responsibility Principle — A class should have only ONE reason to change. One class = one job. Bad: UserService handling login + sending emails + generating reports. Good: Separate AuthService, EmailService, ReportService.
OOpen/Closed Principle — Open for extension, closed for modification. Add new functionality by extending, not by modifying existing code. Use inheritance or composition instead of editing the original class.
LLiskov Substitution Principle — Objects of a subclass should be replaceable with objects of the parent class without breaking the program. If Bird.fly() exists, Penguin extending Bird and crashing on fly() violates LSP.
IInterface Segregation Principle — Don't force a class to implement interfaces it doesn't use. Split large interfaces into smaller, specific ones.
DDependency Inversion Principle — High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces). Your controller depends on IUserService interface, not on a specific UserService implementation — so you can swap it.
How to say it in an interview

"In my Darwinbox project I applied SRP — separate controllers, services, and models. The ResilientAIService demonstrates OCP — I added new model fallbacks without modifying the core service logic. DIP is why I pass services as dependencies rather than instantiating them inside controllers — it makes testing much easier."

Creational Patterns
SingletonOnly one instance of a class exists. Used for: DB connections, config managers, logger. In Node.js, module caching effectively makes every require() a singleton.

The Golden Rule

// Abstract class — IS-A relationship abstract class Shape { String color; abstract double area(); // must implement void display() { System.out.println("Color: " + color); } // shared } // Interface — CAN-DO relationship interface Printable { void print(); } interface Saveable { void save(); } class Report extends Document implements Printable, Saveable { // multiple interface implementation — allowed! }
How to say it in an interview

"Abstract class is for IS-A with shared code — Dog IS-A Animal and shares the eat() implementation. Interface is for CAN-DO — Bird and Airplane both CAN-DO fly, but they're completely unrelated classes. I prefer interfaces because they enable multiple inheritance and loose coupling — my Node.js services depend on interface-like contracts, not concrete implementations."
